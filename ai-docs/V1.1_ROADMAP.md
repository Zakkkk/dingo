## 2. Null Safety Operator (`?.`) ✅ COMPLETE

### Status: SHIPPED (2025-12-01)

Safe navigation operator implemented as part of AST migration.

### Feature Description

Safe navigation operator for chained property/method access that short-circuits on nil:

```dingo
// Instead of:
var name string
if user != nil && user.Profile != nil {
    name = user.Profile.Name
}

// Write:
name := user?.Profile?.Name ?? "Anonymous"
```

### Rationale for Deferral

- **Not MVP Critical:** Developers can use explicit nil checks (verbose but correct)
- **Depends on Other Work:** Benefits from AST infrastructure improvements
- **P1 Priority:** Important but not blocking basic functionality

### Implementation Approach

#### Option A: Preprocessor-based (Quick Implementation)

```
Time Estimate: 1-2 weeks
Pros: Faster to implement, fits current architecture
Cons: Complex regex patterns, harder to maintain
```

Implementation Steps:
1. Create `safe_nav.go` preprocessor
2. Parse `?.` sequences with balanced parenthesis tracking
3. Generate nil check chains
4. Integrate with null coalescing (`??`)

#### Option B: AST-based (Recommended)

```
Time Estimate: 2-3 weeks
Pros: Cleaner implementation, better error messages, easier to extend
Cons: Requires AST infrastructure work first
```

Implementation Steps:
1. Create `SafeNavExpr` AST node
2. Parse in Dingo pre-parser
3. Transform to Go nil-check chains in codegen
4. Generate source mappings

### Code Generation Pattern

```dingo
// Input:
result := user?.Profile?.Settings?.Theme

// Generated Go:
var result string
if user != nil {
    if user.Profile != nil {
        if user.Profile.Settings != nil {
            result = user.Profile.Settings.Theme
        }
    }
}
```

For method calls:
```dingo
// Input:
length := str?.ToUpper()?.Length()

// Generated Go:
var length int
if str != nil {
    tmp1 := str.ToUpper()
    if tmp1 != nil {
        length = tmp1.Length()
    }
}
```

### Success Criteria

- [ ] Property access chains work (`a?.b?.c`)
- [ ] Method call chains work (`a?.Method()?.b`)
- [ ] Combines with null coalescing (`a?.b ?? default`)
- [ ] Works with slice/map indexing (`arr?[0]?.field`)
- [ ] Type inference works for result type
- [ ] Source maps accurate
- [ ] Performance: No overhead when value is non-nil
- [ ] Test coverage: 90%+ for safe_nav.go

### Estimated Effort

- **Preprocessor approach:** 1-2 weeks
- **AST-based approach:** 2-3 weeks (recommended if AST migration in progress)

## 3. Null Coalescing Parser Refinement ✅ COMPLETE

### Status: SHIPPED (2025-12-01)

Null coalescing operator fully implemented with AST-based parsing.

### Current State

**Implementation:** `null_coalesce_ast.go`
**Status:** Complete - all expression patterns supported

### Known Parser Issues

1. **Complex Left-Hand Expressions**
   - Issue: Fails to parse nested expressions correctly
   - Example: `(getValue())?.field ?? default`
   - Cause: Regex-based parsing struggles with balanced parentheses

2. **Method Chain Interaction**
   - Issue: Interaction with method chains causes parsing errors
   - Example: `obj.Method1().Method2() ?? fallback`
   - Cause: Greedy matching in regex patterns

3. **Nested Null Coalescing**
   - Issue: `a ?? b ?? c` may not parse correctly
   - Expected: Right-associative (`a ?? (b ?? c)`)
   - Cause: Pattern matching order issues

### Refinement Approach

#### Quick Fix (Regex Refinement) - 3-5 days

1. Improve parenthesis balancing in regex patterns
2. Add lookahead for method chains
3. Handle nested `??` with explicit grouping
4. Add more comprehensive test cases

#### Complete Fix (AST-based) - Included in AST Migration

If doing AST migration, null coalescing should be migrated together:

1. Create `NullCoalesceExpr` AST node
2. Parse with proper precedence handling
3. Transform to Go conditional expressions

### Success Criteria

- [ ] Simple cases work: `a ?? b`
- [ ] Chained cases work: `a ?? b ?? c`
- [ ] Complex expressions work: `(func())?.field ?? default`
- [ ] Method chains work: `obj.A().B() ?? fallback`
- [ ] Combines with safe navigation: `a?.b ?? c`
- [ ] All existing tests pass (no regressions)
- [ ] 10+ new edge case tests added

### Estimated Effort

- **Regex refinement only:** 3-5 days
- **Full AST migration:** 3-4 days (as part of P2 migration phase)

## 8. V1.2 Roadmap - Deferred Edge Cases

**Target Release:** Q3 2025 (estimated 2-3 months after v1.1)
**Status:** Planning

### Overview

v1.2 focuses on resolving complex edge cases identified during v1.1 AST migration that were deemed non-blocking for general use. These improvements target advanced use cases and enhance type inference capabilities.

### Deferred Items from v1.1

#### Case 2: Complex Multi-Processor Integration

**Problem Description:**
Sequential text-based processing can lose context between processors, affecting complex nested expressions like:
```dingo
let result = items.filter((x) => x?.value ?? 0 > 5)
```

**Technical Challenges:**
- Position drift across multiple transformations
- Context loss between sequential processors
- Nested expression boundary detection
- Order-dependent processing constraints

**Solution Approach:**
1. Expression tree pre-analysis before transformation
2. Identify all operators (`?.`, `??`, `?`, `=>`) in single pass
3. Process innermost expressions first
4. Expand outward with marker-based position tracking

**Implementation Plan:**
- Week 1-2: Design expression tree analysis framework
- Week 3-4: Implement order-independent expression identification
- Week 5-6: Integrate with existing processor pipeline
- Week 7: Testing and edge case validation

**Effort Estimate:** 5-7 days
**Priority:** P2 (affects rare edge cases only)
**Impact:** Improves reliability of complex nested expressions

**Reference:** `ai-docs/sessions/20251201-172045-ast-complete/edge-case-analysis.md` - Case 2

---

#### Case 3: Safe Navigation Type Inference

**Problem Description:**
Safe navigation requires explicit type annotations when types cannot be inferred:
```dingo
// Requires: let user: *User = getUser()
// Cannot infer: let user = getUser()
user?.profile?.name
```

**Technical Challenges:**
- No cross-file type resolution in current TypeDetector
- Cannot infer function return types
- Method return types require full type checking
- No integration with go/types at preprocessor stage

**Solution Approach:**
Integrate go/types for full type inference:

1. **Package-Level Type Cache:**
   ```go
   type TypeResolutionCache struct {
       varTypes    map[string]string  // "user" -> "*User"
       funcReturns map[string]string  // "getUser" -> "*User"
   }
   ```

2. **go/types Integration:**
   - Run go/parser on preprocessed code
   - Use go/types.Config with importer
   - Build type map for all identifiers
   - Use during safe nav transformation

3. **Cross-File Resolution:**
   - Scan all package files before transformation
   - Build package-level type registry
   - Support struct field types across files

**Implementation Plan:**
- Day 1: Integrate go/types into SafeNavASTProcessor
- Day 2: Build package-level type cache
- Day 3: Handle function return types
- Day 4-5: Cross-file resolution and testing

**Effort Estimate:** 4-5 days
**Priority:** P1 (improves developer experience significantly)
**Impact:** Eliminates need for explicit type annotations in most cases

**Reference:** `ai-docs/sessions/20251201-172045-ast-complete/edge-case-analysis.md` - Case 3

### Additional v1.2 Improvements

#### Multi-Statement Null Coalescing (Case 5)

**Current Limitation:** Only one `??` per line processed
**Impact:** Statements like `a := x ?? 0; b := y ?? 1` only process first `??`
**Solution:** Statement splitting before processing
**Effort:** 2-3 days
**Priority:** P2

#### Lambda Error Propagation (Case 1)

**Current Limitation:** Error propagation inside lambda bodies may fail
**Example:** `items.map((x) => readFile(x)?)`
**Solution:** Recursive body processing before lambda wrapping
**Effort:** 2-3 days
**Priority:** P1
